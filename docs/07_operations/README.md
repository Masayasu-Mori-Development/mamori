# 運用設計

システム運用に関する設計書を管理するディレクトリです。

## ファイル構成

- **operations-overview.md**: 運用体制の全体方針
- **deployment-guide.md**: デプロイ手順書
- **monitoring-operations.md**: 監視運用手順
- **backup-restore.md**: バックアップ・リストア手順
- **incident-response.md**: インシデント対応フロー
- **maintenance-plan.md**: 定期メンテナンス計画
- **runbook.md**: 運用手順書（Runbook）

## 運用体制

### 責任者

| 役割 | 責任範囲 |
|------|---------|
| **プロダクトオーナー** | 機能優先順位、リリース判断 |
| **テックリード** | アーキテクチャ、技術選定 |
| **DevOpsエンジニア** | インフラ、CI/CD、監視 |
| **開発者** | 機能実装、バグ修正 |

### オンコール体制

- **営業時間内**: Slackで即時対応
- **営業時間外**: 重大障害のみオンコール対応

## デプロイフロー

### 1. 開発環境（dev）

- **タイミング**: 機能実装完了後、随時
- **承認**: 不要
- **ロールバック**: 必要に応じて実施

### 2. ステージング環境（staging）

- **タイミング**: 週1回（毎週水曜日）
- **承認**: テックリード承認必須
- **テスト**: E2Eテスト実施

### 3. 本番環境（production）

- **タイミング**: 月2回（第2・第4火曜日）
- **承認**: プロダクトオーナー + テックリード承認必須
- **事前準備**: リリースノート作成、ロールバック手順確認
- **実施時間**: 営業時間外（21:00〜23:00）

## 監視・アラート

### CloudWatch監視項目

| 項目 | 閾値 | アラート通知先 |
|------|------|--------------|
| **API エラー率** | > 5% | Slack #alerts |
| **API レスポンスタイム** | > 1秒（95%ile） | Slack #alerts |
| **RDS CPU使用率** | > 80% | Slack #alerts |
| **RDS ストレージ残量** | < 20% | Slack #alerts |
| **ECS タスク異常終了** | 発生時 | Slack #alerts |

### ログ管理

- **保存期間**: 90日間
- **ログレベル**:
  - 本番環境: INFO以上
  - ステージング環境: DEBUG以上
  - 開発環境: DEBUG以上

## バックアップ戦略

### データベース（RDS）

- **自動バックアップ**: 毎日深夜2:00（保存期間: 7日間）
- **手動スナップショット**: リリース前に取得（保存期間: 30日間）
- **リストアテスト**: 月1回実施

### ファイルストレージ（S3）

- **バージョニング**: 有効化
- **ライフサイクルポリシー**:
  - 30日後: Glacier移行
  - 1年後: 削除

## インシデント対応

### 重大度レベル

| レベル | 定義 | 対応時間 |
|--------|------|---------|
| **P0（緊急）** | 全ユーザーに影響、サービス停止 | 即時対応 |
| **P1（高）** | 一部ユーザーに影響、機能停止 | 1時間以内 |
| **P2（中）** | 一部機能に影響、回避策あり | 4時間以内 |
| **P3（低）** | 軽微な問題、影響小 | 翌営業日対応 |

### 対応フロー

1. **検知**: 監視アラート or ユーザー報告
2. **トリアージ**: 重大度レベルの判定
3. **対応**: 調査・修正・リリース
4. **事後対応**: ポストモーテム作成
