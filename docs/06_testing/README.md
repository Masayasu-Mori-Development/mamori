# テスト仕様

テスト戦略とテスト仕様書を管理するディレクトリです。

## ファイル構成

- **testing-strategy.md**: テスト戦略の全体方針
- **unit-test-spec.md**: 単体テスト仕様とカバレッジ目標
- **integration-test-spec.md**: 統合テスト仕様
- **e2e-test-spec.md**: E2Eテスト仕様（Cypress/Playwright）
- **performance-test-spec.md**: パフォーマンステスト仕様
- **test-data-management.md**: テストデータ管理方針

## テストレベル

### 1. 単体テスト（Unit Test）

**目標カバレッジ**: 80%以上

| 対象 | フレームワーク | 実行コマンド |
|------|---------------|-------------|
| **フロントエンド** | Jest + React Testing Library | `npm run test` |
| **バックエンド** | JUnit 5 + MockK | `./gradlew test` |

### 2. 統合テスト（Integration Test）

**目的**: 複数コンポーネント間の連携を検証

| 対象 | 内容 |
|------|------|
| **API統合テスト** | コントローラー → サービス → リポジトリの連携 |
| **データベーステスト** | 実際のDBを使った永続化処理の検証 |

### 3. E2Eテスト（End-to-End Test）

**目的**: ユーザー視点でのクリティカルパスを検証

**対象シナリオ**:
- ログイン → 献立作成 → 保存
- 献立一覧 → 詳細表示 → 編集
- 献立 → 栄養計算 → PDF出力

**ツール**: Cypress または Playwright

### 4. パフォーマンステスト

**目的**: 応答速度と負荷耐性を検証

| 項目 | 目標値 |
|------|--------|
| **APIレスポンスタイム** | < 200ms（95パーセンタイル） |
| **同時接続数** | 500ユーザー以上 |
| **Lighthouse スコア** | 90以上 |

## テストデータ管理

### Fixture

テストデータはFixtureファイルで管理します。

```
tests/
├── fixtures/
│   ├── users.json          # テストユーザーデータ
│   ├── facilities.json     # テスト施設データ
│   └── menus.json          # テスト献立データ
```

### テストDB

- **開発環境**: Docker Compose で PostgreSQL を起動
- **CI環境**: GitHub Actions 上で PostgreSQL コンテナを起動

## CI/CDでのテスト実行

### プルリクエスト時

```yaml
- Lint チェック
- 単体テスト実行
- カバレッジレポート生成
```

### マージ時

```yaml
- 単体テスト実行
- 統合テスト実行
- E2Eテスト実行（クリティカルパスのみ）
- デプロイ
```

## 更新ルール

1. **新機能追加時**: 対応する単体テスト・統合テストを必ず追加
2. **バグ修正時**: バグを再現するテストを追加してから修正
3. **E2Eテストは最小限**: クリティカルパスのみ実装（実行時間を考慮）
4. **テストの可読性**: テストコードも可読性を重視、日本語コメント推奨
